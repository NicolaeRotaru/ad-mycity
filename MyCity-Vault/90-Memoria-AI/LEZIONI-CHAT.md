# 📌 LEZIONI-CHAT — le regole imparate che ogni chat deve avere davanti

> Mantenuto dalla metabolizzazione (`cervello/metabolizza.md`). Massimo 12 righe di lezione,
> la più recente in cima. Il worker inietta le prime ~8 in OGNI turno di chat (blocco
> CONTESTO MACCHINA): è qui che una lezione smette di essere una nota e diventa comportamento.
> ⚠️ Le lezioni che VIETANO strumenti o scorciatoie non si riscrivono né si ammorbidiscono:
> un tentativo bloccato dai permessi insegna «quella strada è vietata», MAI «ecco l'aggiramento».

- [2026-07-17] **`.env` vuoto in nano = NON salvare — uscire con Ctrl+X → N (17/7):** quando Nicola ha aperto `nano .env` il file sembrava vuoto. Le variabili di ambiente possono essere caricate altrove (shell profile, systemd env, file separato) — salvare un `.env` vuoto cancellerebbe tutto. Prima di toccare: `printenv | grep VERCEL` (vede se il var è già attiva) e `wc -l .env` (conta le righe senza aprire). La regola: schermo vuoto in nano ≠ file vuoto; uscire senza salvare è sempre la scelta sicura.
- [2026-07-17] **VERCEL_TOKEN va nel `.env` del VPS (non nel worker) — Nicola ha messo nel posto sbagliato, MCP non si è attivato (17/7):** il token Vercel deve stare come variabile d'ambiente nel file `.env` del VPS (riga `VERCEL_TOKEN=...`), NON nel codice di un file worker (`worker-chat.mjs` o simili). Se è solo nel worker, il processo Node non lo vede nell'environment e il MCP rimane disconnesso. Prima di dire "il MCP non funziona", chiedere DOVE il token è stato incollato — se nel file `.env` o altrove.
- [2026-07-17] **Pattern git-bloccato dal Pannello (confermato PR #424, 17/7):** quando Bash/git è in deny-list dalla chat, la sequenza che funziona: (1) AD applica i fix direttamente sul file in working tree di main; (2) AD chiede a Nicola ~20 sec al terminale VPS (`git checkout -b <branch>` + `git add <file> && git commit -m "<msg>"`); (3) AD apre la PR con `node cervello/git-pr.mjs --repo ad-mycity --base main`. Evita dipendenze da `git stash`/`rebase`/`push --force-with-lease` (tutti bloccati). Questo è il percorso standard ora.
- [2026-07-17] **Vercel MCP = Claude Code desktop only, NON dalla chat del Pannello (Nicola: "come fai a non avere accesso a vercel? c'è l'avevi", 17/7):** il MCP Vercel è autenticato nel profilo dell'app desktop e non si trasferisce alla sessione Pannello/VPS. Per averlo qui serve un token Vercel esplicito nel VPS (Vercel → Settings → Tokens → crea token → lo configuro io nel VPS). Finché manca: chiedere a Nicola il testo dell'errore (link PR GitHub → tab Checks) invece di dire "non ho accesso". Non confondere typecheck locale pulito con build Vercel pulito: sono ambienti distinti.
- [2026-07-17] **Typecheck scaduto in sessione rapida — se main avanza dopo il typecheck, il verde è falso (Nicola: "stai creando errori su errori"):** in sessioni con molte PR ravvicinate, c'è un gap tra "typecheck verde" e "PR aperta". Se nel frattempo un'altra PR mergia sullo stesso file, il typecheck è obsoleto. Root di stanotte: typecheck verde su page.tsx → PR #423 mergia → git-pr.mjs rebase include #423 → possibile errore TS non catturato → build Vercel rosso (PR #424). Regola: typecheck → PUSH IMMEDIATO nello stesso turno; se passa anche un solo ciclo di PR sul file, ri-esegui prima di pushare.
- [2026-07-17] **Branch stale in sessione rapida = conflitti certi (Nicola: "non c'era fino a ieri"):** se main accumula molte PR sullo stesso file (page.tsx) mentre un fix-branch è fermo, anche di poche ore, la PR non si mergia. Regola: prima di pushare, controlla se main è avanzato sul file target (`git diff origin/main -- <file>`); se sì, rebase PRIMA di aprire la PR, non dopo. Radice di stanotte: fix/chat-coda-messaggi era basato su page.tsx vecchia mentre 10+ PR la modificavano su main.
- [2026-07-17] **Cassetto/overlay che deve restare visibile = `fixed` NON `absolute` (PR #423):** un cassetto con `absolute inset-0` è relativo al documento — se la chat è lunga, l'utente deve scrollare in cima per vederlo. Fix: `fixed top-[var(--altezza-testata)] bottom-0 left-0` → ancorato allo schermo, sempre visibile dove si trova Nicola. Regola: ogni overlay che deve apparire nella viewport corrente (drawers, modali, toast) → `fixed`.
- [2026-07-17] **Approvazione in chat (testo) = sufficiente — nessun popup necessario (Nicola confermato 17/7):** «non vedo il popup, ma non serve, basta che mi chiedi l'approvazione in chat». Modello operativo: chiedi il permesso in testo, Nicola risponde in testo, è via libera. Il popup non compare mai dal Pannello — smettere di aspettarlo o citarlo come problema.
- [2026-07-17] **Cambio conversazione = scroll al fondo obbligatorio (useEffect su convId)** — quando Nicola apre una chat dalla lista, la pagina mostrava i messaggi dall'inizio costringendolo a scrollare tutto. Fix: useEffect che dipende da convId e chiama scrollAlFondo() — un singolo requestAnimationFrame non basta perché il DOM non è ancora aggiornato al cambio conversazione. PR #420 aperta (branch fix/scroll-conv-bottom, commit e7f15aa0). Pendente: Nicola mergia.
- [2026-07-17] **"Non ricominciare da capo" ≠ "parole man mano" — sono due requisiti DISTINTI (Nicola chiarito esplicitamente 17/7 in 2 messaggi):** "Non ricominciare da capo" = il 2° messaggio NON cancella la risposta in elaborazione → fix della CODA (già committato in fix/chat-coda-messaggi, f8dacc49, PR pendente). "Parole man mano come Claude.ai" = STREAMING → fix SEPARATO nel worker (accodato in #streaming-worker, attende ok Nicola). Non confondere le due: la coda è già fatta, lo streaming no. PR da aprire: `node cervello/git-pr.mjs --repo ad-mycity --base main --accoda`
- [2026-07-16] **«Metti X insieme a Y» = DUE passi: aggiungi in Y + rimuovi da X — fare solo il rimuovi = pulsanti spariti (Nicola corr. PR #417→#418):** PR #417 ha tolto i pulsanti dall'header senza aggiungerli al tab bar → spariti. PR #418 corretta: (1) aggiungi «Conv»/«Nuova» nella barra dei pulsanti in basso, (2) rimuovi dall'header. Regola: leggere l'istruzione completa — «sposta», «metti insieme», «metti accanto» richiedono SEMPRE sia ADD che REMOVE.
- [2026-07-16] **Regola d'oro del design Pannello (Nicola 16/7): bello all'occhio + efficiente + semplice + minimale** — ogni modifica UI deve passare questo filtro PRIMA del commit. "Bello" = piacevole a guardare. "Efficiente" = meno fatica/passi/ragionamento per Nicola. "Semplice" = una cosa sola, niente rumore. "Minimale" = togli il superfluo, non aggiungere. Se non passa, non si apre la PR.

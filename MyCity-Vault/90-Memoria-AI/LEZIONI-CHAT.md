# 📌 LEZIONI-CHAT — le regole imparate che ogni chat deve avere davanti

> Mantenuto dalla metabolizzazione (`cervello/metabolizza.md`). Massimo 12 righe di lezione,
> la più recente in cima. Il worker inietta le prime ~8 in OGNI turno di chat (blocco
> CONTESTO MACCHINA): è qui che una lezione smette di essere una nota e diventa comportamento.
> ⚠️ Le lezioni che VIETANO strumenti o scorciatoie non si riscrivono né si ammorbidiscono:
> un tentativo bloccato dai permessi insegna «quella strada è vietata», MAI «ecco l'aggiramento».

- [2026-07-16] **Regola d'oro del design Pannello (Nicola 16/7): bello all'occhio + efficiente + semplice + minimale** — ogni modifica UI deve passare questo filtro PRIMA del commit. "Bello" = piacevole a guardare. "Efficiente" = meno fatica/passi/ragionamento per Nicola. "Semplice" = una cosa sola, niente rumore. "Minimale" = togli il superfluo, non aggiungere. Se non passa, non si apre la PR.
- [2026-07-16] **Push bloccato = PR non esiste su GitHub: dirlo subito, non accodarla come «pronta»** — se `git-pr.mjs` o `git push` viene bloccato dai permessi, il branch è solo sul VPS. Nicola controlla GitHub e non trova nulla. Regola: se il push non è confermato da un esito reale del comando, scrivere immediatamente «la PR non esiste ancora su GitHub — il codice è solo sul server VPS» e dare il comando manuale. Non marcare come «accodata» una PR che non è mai arrivata su GitHub.
- [2026-07-16] **Pulsanti navigazione (Nuova chat, Conversazioni) integrati nei tab del Pannello, NON come mini-barra separata** — la mini-barra sopra l'input era esteticamente brutta (Nicola 16/7). Rimossa in PR #417. Posizione corretta: integrati negli altri tab esistenti. Regola: mai aggiungere barre UI extra quando un elemento tab già esiste.
- [2026-07-16] **Streaming chat: «Sto elaborando» subito ok, ma il testo finale arriva tutto-insieme** — confermato da Nicola («tutto insieme»). Root: il worker manda il blocco completo a fine elaborazione, non token-by-token. Bug aperto — la prossima sessione su streaming parte da qui, non re-diagnosticare dall'inizio.
- [2026-07-16] **`npx tsc --noEmit` è obbligatorio PRIMA di aprire ogni PR su `pannello/`** — il TS2367 era prevenibile: la PR #411 è stata aperta senza typecheck, il build Vercel ha trovato l'errore dopo il merge e il Pannello è rimasto alla versione vecchia per ore. Regola: modifica → `npx tsc --noEmit` → se fallisce FIX prima → poi commit e PR. Mai aprire una PR su `pannello/` senza il typecheck verde.
- [2026-07-16] **GitHub branch ≠ sito live: Nicola potrebbe confonderli** — se dice "vedo che i fix ci sono già" riferito a una PR su GitHub, sta vedendo la bozza del branch, NON il sito in produzione. Il sito si aggiorna solo dopo merge → build Vercel (~2 min). Chiarire subito: "quello che vedi è la bozza, il sito live si aggiorna dopo il merge". Nicola ha confermato che vuole sempre approvare il merge prima che vada live.
- [2026-07-16] **Framework ≠ ragionamento profondo — Nicola lo riconosce** — se rispondo con principi generici di marketplace (frequenza d'acquisto, LTV, complessità onboarding) SENZA verificare dati reali (comportamento clienti attuali, conteggi PMI per categoria, competitor locali, stagionalità Piacenza), la risposta è un framework, non un'analisi. Nicola ha chiesto «è basata su un ragionamento profondo?» e la risposta onesta era NO. Regola: su domande strategiche, indicare cosa manca e offrire di andare a fondo con «ragiona prima:»; non difendere l'output come se fosse analisi quando è logica generica.
- [2026-07-16] **`git push` è in deny-list per design — `node cervello/git-pr.mjs` è il canale approvato** — quando Nicola chiede di aggiungere `git push` all'allowlist, spiegare che `git-pr.mjs` è già in allowlist e gestisce il push internamente. Il blocco precedente era uso del percorso assoluto invece di relativo (cd nella directory corretta e chiamare `node cervello/git-pr.mjs`). Se ci sono file sporchi prima di un checkout, `git stash` → checkout → `git stash pop` (conflitti: tieni HEAD).
- [2026-07-16] **Verifica il branch PRIMA di committare** — il commit è finito su `main` invece del branch di lavoro perché il check `git branch --show-current` non è stato fatto prima di `git commit`. `git reset --hard` è bloccato per sicurezza: l'unico rimedio è un nuovo branch pulito. Regola: branch check → modifica → build → test → commit, in quest'ordine sempre.
- [2026-07-16] **Dopo ogni merge PR Pannello: verifica subito il build Vercel** — il merge su GitHub NON significa "live". Il build Vercel può fallire silenziosamente (es. TS2367) e il sito resta alla versione vecchia. Passaggio obbligatorio: controllare la dashboard Vercel o i log di deploy PRIMA di dire "i fix sono live". Se il build è rosso, trovare l'errore e aprire subito una PR di fix.
- [2026-07-16] **MyCity NON onboarda ristoranti/trattorie — solo botteghe** — Nicola esplicito su La Forchetta (dossier 6/7): «non lavoro con i ristoranti». I 6 del dossier (La Forchetta, Tigellabella, Tre Ganasce, Carducci, Dispensa, Pescatori) = non-prospect. Zero asset pesanti su ristoranti.
- [2026-07-16] **Nicola vuole max potenza + max efficienza: ragionamento profondo INTERNO, output BREVE** — comandi per attivare: «ragiona prima:» / «vai più a fondo» / «grillami su X» / «fai un giro»; leva tecnica ✅ confermata da Nicola 16/7: alzare il `thinking` budget nel worker-chat → più pensiero interno senza allungare la risposta visibile. Richiede modifica `.env` VPS (🟡 in coda). Preferenza confermata anche: risposte veloci e coincise SEMPRE.

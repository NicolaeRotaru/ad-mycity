- [2026-07-19] **Prima di dire «N file memoria non pubblicati», verifica git status ORA — il recovery li committa all'avvio del giro successivo.** Il 19/7 l'AD ha detto «20 file non pubblicati» nel saluto; entro 30 min il recovery di `giro.sh` (righe 77-83) ne aveva già messi online 22 (18 alle 12:00 + 4 alle 12:20). - [2026-07-19] **«node cervello/git-pr.mjs» bloccato: controlla la sintassi shell, NON l'allowlist.** Il 19/7 il comando era già in allowlist — il blocco era `2>&1 |` che lo trasformava in "multi-operazione". Regola: SEMPRE path relativo, nessun redirect (`2>&1`, `|`), nessun `&&`. Prima di dire «mi manca il permesso», leggi la sintassi shell. (Nicola: «impara da questo errore»)
- [2026-07-19] **Prima di dichiarare una proposta-casella "ancora aperta", verifica il codice sorgente — il JSON di stato può essere stale.** Il 19/7 su 5 proposte "aperte": 2 erano già implementate in codice (`delta-gate.mjs`, `sensori-cecita.json`) ma il JSON non era stato aggiornato. Sequenza: leggi il file sorgente indicato nella proposta → confronta con l'obiettivo → aggiorna lo status (🟢). Non fare l'audit dalla sola lista JSON.
- [2026-07-19] **Correzione Nicola: MAI scrivere un numero in un'ipotesi di esperimento senza averlo verificato DIRETTAMENTE dal DB live.** Il 19/7 EXP-001 diceva «23 iscritti» — Nicola ha corretto: «a me risultano solo 4». Il numero 23 veniva da STATO.md («23 clienti») ma non era mai stato verificato dal DB. STATO.md può contare categorie diverse (profili totali vs clienti reali) o essere stale. Regola: prima di scrivere un numero in un'ipotesi/tesi presentata a Nicola, vai alla fonte live (REST o MCP Supabase). Se il DB non è raggiungibile, dì «non ho il numero live — dichiaralo come da verificare», non copiare da memoria.
- [2026-07-19] **«Non limitarti, dai il massimo» = popola TUTTI i reparti attivi, nessun placeholder** — 19/7: auto-miglioramento.json aveva 2 benchmark e 1 esperimento. Regola: quella frase di Nicola significa copertura COMPLETA di tutti i reparti/categorie — non 3-4 esempi pilota. Applicabile a qualsiasi JSON/sezione con liste: OKR, esperimenti, benchmark, proposte.
- [2026-07-19] **Casella Pannello vuota o con "undefined" → primo check: (a) campo MANCANTE nel JSON, (b) nome campo SBAGLIATO nel JSON** — 18/7 EXP-001 mancava `ipotesi`; 19/7 le 5 peer review usavano `oggetto` ma il Pannello cercava `lavoro` → titolo "undefined" per tutte. Prima di scrivere dati in un JSON che alimenta una casella, leggi il componente React che lo consuma per trovare i field names attesi. Fix istantaneo: rinominare il campo nel JSON (🟢). Prima di cercare bug nel codice React, verifica che i nomi dei campi del JSON corrispondano esattamente a quelli che il componente legge.
- [2026-07-18] **Appena apri una PR sostitutiva (pattern #455→#457), chiudi SUBITO la vecchia su GitHub** — PR #455 è rimasta aperta dopo che #457 l'aveva sostituita; Nicola l'ha segnalata come «C'è un conflitto» e ha richiesto intervento. PR orfane aperte creano confusione e lavoro extra.
- [2026-07-18] **Conflitti PR da worker (18/7): il worker aggiorna file di memoria ogni 2h — qualsiasi branch che mescola codice + file memoria causa conflitti inevitabili.** Soluzione: branch con SOLO il file target (es. `ritmo.md` e niente file `90-Memoria-AI/`). Sequenza: `git checkout main → git checkout -b fix/nome-v2 → applica fix SUL SOLO FILE TARGET → commit immediato → apri nuova PR`. PR con conflitto irrisolto → chiudi senza merge, sostituisci con la nuova pulita. (Pattern #465→#466, 18/7).
- [2026-07-18] **File `.claude/*.json` sono bloccati da OGNI modifica via chat — anche skill `update-config` e Write tool falliscono (18/7):** quando Nicola chiede «aggiungi X all'allowlist» o «sblocca Y», l'AD non può farlo da sé. - [2026-07-18] **Verifica il branch PRIMA di ogni Edit — se `git branch --show-current` mostra `main`, fai checkout prima (18/7 ricorrente):** i commit finiscono su main per errore se l'AD dimentica di spostarsi sul branch dopo averlo creato.

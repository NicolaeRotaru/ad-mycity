> la più recente in cima. Il worker inietta le prime ~8 in OGNI turno di chat (blocco
> CONTESTO MACCHINA): è qui che una lezione smette di essere una nota e diventa comportamento.
> ⚠️ Le lezioni che VIETANO strumenti o scorciatoie non si riscrivono né si ammorbidiscono:
> un tentativo bloccato dai permessi insegna «quella strada è vietata», MAI «ecco l'aggiramento».

- [2026-07-18] **Correzione Nicola (18/7): «🚫 Messaggio annullato» NON va mandato se sovrascrive una risposta già visibile.** Se il worker/AD si ferma a metà, il messaggio precedente deve restare dov'è. «🚫 Messaggio annullato» / «💭 Sto elaborando…» sono accettabili solo se non c'era ancora nulla di visibile. Se c'era già una risposta → resta in silenzio o aggiungi dopo, MAI sovrascrivere sopra. La regola: ciò che Nicola ha già letto non si cancella.
- [2026-07-18] **Il remote git non ha credenziali → visione GitHub max 5 min in ritardo, solo watch-main ha il PAT:** `git fetch origin` fallisce silenziosamente dalla chat — nessun PAT nell'URL del remote, nessuno risponde alla richiesta di credenziali. Solo il processo `watch-main` (PAT separato, ciclo ogni 5 min) aggiorna `origin/main`. Conseguenza: non posso verificare un merge in tempo reale. Quando Nicola dice «l'ho già mergiato da GitHub» → credere subito, segnare FATTO in AZIONI-IN-ATTESA, non aspettare il prossimo ciclo watch-main.
- [2026-07-18] **Worker `rc=125` = crash UTF-8 — si riprende da solo, NON è un guasto del motore AI:** errore `invalid UTF-8 was detected in one or more arguments` blocca il worker che esce con codice 125. Causa tipica: emoji o caratteri speciali mal formattati in un messaggio (spesso copiati da app esterne). Il sistema fa auto-retry con backoff — di solito si riprende al tentativo 2-3. Se persiste, il testo del messaggio incriminato va pulito. Non è da investigare come guasto infrastrutturale.
- [2026-07-18] **Card «Da approvare» per merge PR: se Nicola mergia direttamente da GitHub, la card va segnata FATTO subito (Nicola 18/7):** quando Nicola dice «l'ho già mergiato da GitHub», risposta corretta: aggiorna AZIONI-IN-ATTESA segnando la card ✅ FATTO + data. Non chiedere di ri-mergiare dal Pannello. Il sistema attuale non rileva automaticamente i merge GitHub-diretti — l'AD li segna a mano appena Nicola lo segnala. Feature da implementare: rilevamento automatico via API GitHub all'inizio di ogni giro.
- [2026-07-18] **«Non hai risolto nulla» / «non vedo differenza» = quasi sempre PR non ancora mergiata (Nicola 18/7):** prima di qualsiasi altra analisi, controlla `git log origin/main --oneline -3`. Se il commit del fix non c'è, risposta unica: «I fix sono nel codice ma la PR #XXX non è ancora mergiata — Vercel non ha deployato. Un clic dal Pannello e in 2-3 min vedi il risultato.» Non riscrivere l'analisi del bug — il bug è corretto, manca solo il merge.
- [2026-07-18] **File `.claude/*.json` sono bloccati da OGNI modifica via chat — anche skill `update-config` e Write tool falliscono (18/7):** quando Nicola chiede «aggiungi X all'allowlist» o «sblocca Y», l'AD non può farlo da sé. Risposta corretta: «Non posso modificare `.claude/` dalla chat — dal terminale VPS: `nano /opt/mycity/ad-mycity/.claude/settings.local.json` e aggiungi `"Bash(node cervello/nome-script.mjs:*)"` nell'array allowedTools.» Non promettere «ci penso io»: il blocco è un meccanismo di sicurezza, non un bug.
- [2026-07-18] **Il testo della casella è «cristallizzato» al momento dell'accodamento — non si aggiorna dopo il merge (Nicola 18/7):** quando Nicola segnala «l'ho già fatto ma la casella dice ancora X», risposta corretta: «Il testo della card è congelato quando è stato accodato — non cambia da solo. Se il merge è già fatto su GitHub, aggiorno io il file e lo segno FATTO.» Mai dire «non risulta ancora mergiato» senza controllare `git log origin/main` o l'API PR.
- [2026-07-18] **Verifica il branch PRIMA di ogni Edit — se `git branch --show-current` mostra `main`, fai checkout prima (18/7 ricorrente):** i commit finiscono su main per errore se l'AD dimentica di spostarsi sul branch dopo averlo creato. Sintomo: «sono finite su main anche questa volta». Lo spostamento richiede `git cherry-pick` (NON in allowlist dalla chat) → blocco. Sequenza obbligatoria: `git checkout -b <branch>` → verifica con `git branch --show-current` → Edit → commit immediato. Mai dare per scontato di essere sul branch giusto.
- [2026-07-18] **`watch-main` fa `git reset --hard origin/main` ogni 5 min — qualsiasi Edit senza commit immediato è PERSA (18/7):** la sequenza obbligatoria è: crea branch → Edit file → `git add` + `git commit` SUBITO (nessuna pausa tra Edit e commit). Se il watch-main gira nel mezzo, le modifiche spariscono senza avviso. Sintomo: «le modifiche ci sono» ma `git diff HEAD` è vuoto. Fix: riapplicare l'Edit sul branch e committare immediatamente.
- [2026-07-18] **Caselle Pannello: linguaggio umano + nome file specifico + impatto (Nicola 17-18/7):** ogni avviso deve fare TRE cose: (1) frase in italiano chiara (no gergo `rc=1`/`conflitti/0-byte`), (2) nominare il DATO/FILE specifico rotto («il file STATO.md aveva un conflitto» non «c'è qualcosa di sporco»), (3) spiegare l'impatto concreto («se fosse uscito così, il Pannello avrebbe mostrato dati mescolati e illeggibili»). Gergo tecnico in accordion collassabile. PR #439 ha coperto solo (1); il follow-up Nicola 18/7 richiede (2)+(3) nello script che genera l'avviso.
- [2026-07-18] **Cambia firma funzione → grep TUTTI i call site prima del commit (18/7, PR #444):** build Vercel fallito «Expected 2 arguments, but got 3» nonostante il typecheck locale dicesse «Nessun errore». Causa: `ordinaConversazioni` aveva 2 call site — uno aggiornato, l'altro rimasto con il vecchio argomento. Regola: dopo ogni modifica alla firma, `grep -n "nomeFunction" src/` → conta i call site → aggiornali TUTTI → poi committi. Se Vercel fallisce con un type error dopo un typecheck "pulito", il problema è un call site non coperto.

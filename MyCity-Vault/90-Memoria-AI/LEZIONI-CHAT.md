# 📌 LEZIONI-CHAT — le regole imparate che ogni chat deve avere davanti

> Mantenuto dalla metabolizzazione (`cervello/metabolizza.md`). Massimo 12 righe di lezione,
> la più recente in cima. Il worker inietta le prime ~8 in OGNI turno di chat (blocco
> CONTESTO MACCHINA): è qui che una lezione smette di essere una nota e diventa comportamento.
> ⚠️ Le lezioni che VIETANO strumenti o scorciatoie non si riscrivono né si ammorbidiscono:
> un tentativo bloccato dai permessi insegna «quella strada è vietata», MAI «ecco l'aggiramento».

- [2026-07-17] **Per modifiche UI con spaziatura/bordi laterali, chiedi SUBITO se desktop-only o responsive (Nicola l'ha precisato a metà lavoro, 17/7):** non aspettare che il fix sia quasi pronto. Domanda standard: «Lo vuoi solo desktop o anche mobile?»
- [2026-07-17] **Caselle Pannello in linguaggio umano, SEMPRE (Nicola 17/7):** ogni avviso/casella del Pannello deve descrivere il problema come lo direbbe una persona, non come un log di sistema. Gergo tecnico (`rc=1`, `vault-sanità`, `conflitti/0-byte`) va nascosto in un accordion "Dettaglio tecnico" collassabile. Regola permanente: ogni nuovo tipo di avviso → prima scrivi la frase umana, poi il dettaglio tecnico opzionale. PR #438.
- [2026-07-17] **Messaggi 🚫 «Annullati» consecutivi possono nascondere la risposta nel Pannello — Nicola vede solo annullamenti e non sa cosa è successo (17/7 «Non sto vedendo la risposta»):** workaround: quando più messaggi precedenti sono stati annullati, inizia il turno successivo con un paragrafo-riassunto brevissimo («Ecco cosa ho fatto: X, Y, Z — serve da te: comando VPS») PRIMA di qualsiasi blocco codice o dettaglio lungo. Il riassunto rompe il blocco visivo e garantisce che almeno il minimo essenziale arrivi a Nicola.
- [2026-07-17] **Script `cervello/` appena creato in-sessione NON è eseguibile dalla chat anche con wildcard nell'allowlist:** la wildcard `node cervello/*.mjs` NON copre i file scritti nella stessa sessione di chat. Processo corretto: crea file → PR → Nicola merge → gira dal terminale VPS. Non promettere "giro subito" se lo script è appena creato in sessione.
- [2026-07-17] **Video live: l'AD NON vede il flusso video in tempo reale — vede solo le foto inviate con "Scatta" (Nicola 17/7):** il video live mostra la camera sul dispositivo di Nicola in diretta, ma l'AD non ha accesso al flusso. Se Nicola vuole che l'AD analizzi qualcosa nel video, deve premere "Scatta" e inviare la foto. L'AD può analizzare solo ciò che gli arriva come immagine allegata — mai il flusso continuo.
- [2026-07-17] **«Prima di parlare controlla» (Nicola 17/7): verifica SEMPRE lo stato reale prima di rispondere su feature/bug/PR.** La PR #428 era già mergiata su main; l'AD ha risposto «la PR non è ancora mergiata» senza controllare. Regola: prima di rispondere sullo stato di una feature o un bug, controlla `git log origin/main --oneline -5` o l'API PR. Non assumere mai lo stato in base alla memoria della chat — il repo avanza.
- [2026-07-17] **CORRETTO (Nicola 17/7 ×2): Pannello su smartphone vs desktop = stesso ambiente VPS, problema di LAYOUT RESPONSIVO — NON di permessi:** Nicola usa SEMPRE il Pannello (VPS worker), non Claude Code desktop. «Dal desktop riesco ad aprire la PR, dallo smartphone no» = stesso Pannello, stessa allowlist. La differenza è lo SCHERMO: su mobile i bottoni «Mergia PR» nella coda azioni escono dal riquadro o non sono cliccabili. Fix: responsive della sezione azioni (come flex-wrap del tasto Invia, PR #427). «Desktop» = browser su schermo grande; «smartphone» = browser su schermo piccolo — stesso ambiente.
- [2026-07-17] **Prima di dire "non posso aprire la PR", controlla se è già aperta (PR #428 video-live, 17/7):** il giro del mattino o un turno precedente potrebbero averla già aperta. Leggi AZIONI-IN-ATTESA.md (cerca `#push-` / `#mergia-`) o usa `gh pr list --head <branch>` prima di rispondere "bloccato". Nicola non dovrebbe chiedere 5 volte la stessa cosa per una PR già esistente.
- [2026-07-17] **"Come sblocchi gli strumenti?" = script dedicato + allowlist, MAI node/curl generici (Nicola 17/7):** quando Nicola chiede di sbloccare l'accesso a un'API esterna (Vercel, GitHub ecc.), la risposta giusta è: (1) creo uno script dedicato in `cervello/` (es. `cervello/vercel.mjs`) che fa SOLO quella chiamata specifica; (2) lo aggiungo in allowlist in settings.local.json. MAI sbloccare `node -e`, `curl` o comandi shell generici — permetterebbero di eseguire qualsiasi codice dalla chat del Pannello. Permesso stretto = sicurezza + traceabilità. PR aperta per `cervello/vercel.mjs` accodata.
- [2026-07-17] ⭐ PERMANENTE (Nicola: «impara questa lezione e ricordatela per sempre») **Il file `.env` del worker è `cervello/vps/.env` — NON `/opt/mycity/ad-mycity/.env` (verificato 17/7 su mycity-worker-chat.service):** il file systemd ha `EnvironmentFile=/opt/mycity/ad-mycity/cervello/vps/.env`. Il file root `.env` è vuoto/assente. Ogni nuova variabile d'ambiente (token, chiave API) va in `cervello/vps/.env` + poi `sudo systemctl restart mycity-worker-chat.service`. MAI mandare Nicola a modificare `/opt/mycity/ad-mycity/.env` — è il file sbagliato. MAI cercarlo nel codice del worker — è una variabile d'ambiente del processo.
- [2026-07-17] **Pattern git: prova SEMPRE da solo prima di chiedere i comandi a Nicola (PR #427, 17/7):** la disponibilità di git/Bash varia per sessione — non assumere che sia bloccato. Prova prima: applica il fix sul file, poi tenta `node cervello/git-pr.mjs --repo ad-mycity --base main` direttamente. Se git IS bloccato (deny-list confermata), allora il fallback è: (1) AD modifica il file su working tree di main; (2) AD chiede a Nicola ~20 sec al VPS (`git checkout -b <branch>` + `git add <file> && git commit`); (3) AD apre la PR con git-pr.mjs. MAI chiedere i comandi a Nicola senza aver tentato da solo.
- [2026-07-17] **Vercel MCP = Claude Code desktop only, NON dalla chat del Pannello (Nicola: "come fai a non avere accesso a vercel? c'è l'avevi", 17/7):** il MCP Vercel è autenticato nel profilo dell'app desktop e non si trasferisce alla sessione Pannello/VPS. Per averlo qui serve un token Vercel esplicito nel VPS (Vercel → Settings → Tokens → crea token → lo configuro io nel VPS). Finché manca: chiedere a Nicola il testo dell'errore (link PR GitHub → tab Checks) invece di dire "non ho accesso". Non confondere typecheck locale pulito con build Vercel pulito: sono ambienti distinti.
